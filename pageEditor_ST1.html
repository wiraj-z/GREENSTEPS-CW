Home page

Technical Description

HTML Structure

The page is built on semantic HTML5 landmarks. A header element contains the site brand and a nav element for the primary navigation, main wraps all page content, and a footer closes the document with student and site links. The content inside main is divided into section elements, each connected to its own heading through aria-labelledby, giving the page a clear outline: one h1 in the hero, then h2 headings for each section and h3 headings inside the cards. The mission and topic cards are marked up as article elements because each card is a self-contained unit of content that would make sense on its own. Unordered lists present the key action points inside each card. Supporting tags include a skip link anchor for keyboard users, img elements with descriptive alt text, and width, height and loading attributes on images to reduce layout shift and delay loading of off-screen pictures.

CSS Techniques

Two layout systems are used together. Flexbox centres the hero text vertically and horizontally over a full-viewport background image, and also arranges the pair of call-to-action buttons. CSS Grid drives the card layouts: the topic gallery is a four-column grid that collapses to two columns at 900px and a single column at 640px through media queries. Positioning is used in the hero, where the background image is absolutely positioned behind the text and a linear-gradient overlay darkens it so the white heading stays readable. Styling decisions centre on a green palette defined once as CSS variables in the shared stylesheet, generous spacing through consistent padding and gap values, and rounded corners and soft shadows on every card. Hover effects lift cards with a small translateY movement, zoom their images with a transform scale, and reveal a green gradient overlay, all softened with transitions. Keyframe animations add a slow hero zoom, a bouncing scroll hint and staggered fade-in entrances, and a prefers-reduced-motion query switches these off for users who opt out of animation.

Design and Implementation Rationale

Generating cards from a JavaScript array was chosen so that adding a new gallery story means adding one object to the array instead of duplicating blocks of HTML, which reduces errors and keeps the markup short. The dialog role and aria attributes were included because a visually obvious popup is invisible to a screen reader without them. Cards are given a tabindex and respond to Enter and Space as well as clicks, so the gallery is fully usable without a mouse, and the modal can be dismissed either with the close button or by clicking the backdrop, matching the behaviour users expect from dialogs. Using a checkbox for the theme switch means the browser provides the state handling and keyboard support for free, and the CSS-only knob avoids extra images. The consistent card styling, shared colour variables and matching breakpoints keep the gallery visually aligned with the home page.

Accessibility

All images include descriptive alt text (e.g. alt="family cycling in a park on a sunny day, representing sustainable transport"). The statistics bar uses an visually hidden heading so screen readers announce the section without affecting the visual layout. Semantic HTML5 elements provide a logical document structure for assistive technologies.

Gallery Page

Technical Description

The Gallery page follows the same structure as the rest of the website, with a header, main section, and footer. The gallery grid is empty in the HTML because the cards are created dynamically with JavaScript when the page loads. It also includes an accessible modal dialog using role="dialog", aria-modal, and aria-labelledby. The close button is a standard button element, and the dark mode toggle uses a checkbox inside a label, making both features keyboard accessible.

The CSS creates a responsive grid that changes from four columns to two and then one on smaller screens. The modal uses fixed positioning to cover the page and is shown by adding an open class. The dark mode switch is styled with CSS using a pseudo element and the :checked selector. A .dark class changes the modal colours, and transitions make the theme change smooth.

JavaScript handles all the interactivity. Gallery items are stored in an array of objects, and a forEach loop generates each card automatically. Event listeners manage hover effects, mouse clicks, and keyboard input so users can open cards with Enter or Space. The openModal() function displays the selected image, title, and description. Users can close the modal using the close button or by clicking outside it. A change listener on the theme switch uses classList.toggle() to switch the modal between light and dark mode.

Content Page

Technical Description

This page is a long-form article on sustainable transport. After the shared header, the main element opens with a hero holding the h1 title, an intro paragraph and an image. Below it, a nav with an aria-label contains a list of anchor links that jump to the five numbered sections, working as a table of contents. Each topic is a section element with its own id for the anchors to target, using h2 and h3 headings, paragraphs and bullet lists. Images sit in figure tags with figcaption captions, the correct way to pair an image with its caption, and all carry descriptive alt text and lazy loading.

The CSS uses a two-column Grid to place text beside each figure, and a reverse-layout class flips the columns with the order property for a zigzag rhythm. The min() function caps the reading width while keeping padding on small screens, and media queries at 900px and 650px stack the columns and shrink the type. Rounded borders, soft shadows and the shared green palette keep it consistent with the rest of the site, while scroll-margin-top stops headings hiding under the sticky header.

Interactivity comes from HTML and CSS, not JavaScript. Smooth scrolling on the html element makes the table-of-contents and back-to-top links glide rather than jump, hover and focus-visible states give buttons and links clear feedback, and images scale slightly on hover. The mobile menu uses the same CSS checkbox technique as the rest of the site, so the page stays fully functional even without JavaScript.

Challenges and Lessons Learned

One of the main challenges was building a shared template that all four students could use consistently while each page also needed its own embedded CSS. The solution was to strictly separate all shared rules into style.css and only add page-specific rules in the embedded <style> block, using clear comments to mark which rules belong to which scope.

Implementing the Gallery modal was also challenging — populating the correct image and description dynamically from a JavaScript array required careful indexing and DOM manipulation. Debugging in browser DevTools helped resolve an issue where the modal image dimensions were not resetting correctly between clicks.

Accessibility Considerations

Accessibility was built into all three pages from the start. Each page uses semantic landmarks, header, nav, main and footer, with a logical heading order and aria-labels, so screen readers can map the page and users can jump between sections. Every meaningful image has descriptive alt text, decorative layers are hidden with aria-hidden, and the content page pairs images with figure and figcaption.

Colour is handled carefully: the shared green palette was chosen for good contrast, the hero uses a dark overlay so white text stays readable over the photo, and colour is never the only cue, since focus outlines, movement and shape reinforce it for colour-blind users. Every interactive element works by keyboard, with a skip link on each page, visible focus-visible outlines, and hover states matched by focus states. The JavaScript-generated gallery cards were given a tabindex and Enter and Space key support so they behave like real buttons.
