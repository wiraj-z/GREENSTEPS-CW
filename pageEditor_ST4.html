User Profile Page

Technical Description

The page reuses the shared header, navigation, and footer from the team template without modification. Inside <main> I used <section> elements with aria-labelledby pointing at each heading's id, and <article> for the two profile-summary cards so each is understood as a standalone unit of content. A <dl> (description list) holds each profile field as a term/value pair, which is more semantically correct than a plain list for label–value data. Layout uses the Global CSS .grid-2 utility for the two-column cards, which already collapses to one column at the site-wide 900px breakpoint. Page-specific embedded CSS handles the progress bar and the hide/reveal behaviour for the profile-summary section via a single .hidden { display: none; } class toggled by JavaScript. The spec required JavaScript prompts rather than an HTML form, so the main design problem was giving the user feedback despite prompt()/confirm() being blocking and visually disconnected from the page. I addressed this with a visible progress bar and percentage text that update immediately after each step, a summary list that always shows all nine fields and marks missing ones explicitly, and an "Edit Missing Information" button that only re-prompts for empty fields, so returning users aren't asked to repeat information they already gave.

Accessibility

aria-live="polite" is set on the progress text so screen-reader users are told about progress changes automatically without needing to navigate back to that element. The progress track uses role="progressbar" with aria-valuemin, aria-valuemax, and aria-valuenow kept in sync by JavaScript. Every action (start, edit, reset) is a real <button> rather than a clickable <div>, so it is reachable and operable by keyboard by default.

Sitemap Page

Technical Description

The sitemap is built as a single inline <svg> inside <main>, using <a> elements (SVG's native link element) wrapping a <rect> and <text> per page, so each node is a genuinely clickable link rather than a JavaScript click-handler standing in for one. The SVG uses viewBox with preserveAspectRatio="xMinYMin meet" instead of fixed pixel dimensions, so the whole tree scales down on tablet/mobile without needing separate breakpoints. Node styling — fill, stroke, hover and focus states — is done entirely with CSS selectors targeting the SVG elements (.sm-node rect, :hover, :focus-visible) rather than inline SVG attributes, keeping styling in one place and consistent with how the rest of the site handles interactive states. I chose a horizontal, left-to-right tree layout because it scales more predictably on narrow screens than a top-down tree, and grouped each student's Content Page with its own Page Editor and Validation page as children, so the diagram reflects the real navigation paths on the site rather than just the top-level nav bar.

Accessibility

Each node carries an SVG <title>, which browsers expose as an accessible name, and the whole diagram has a top-level <title> and <desc> referenced via aria-labelledby so screen-reader users get a text description of the tree structure before reaching the individual nodes. Every node also has tabindex="0" so the tree is fully keyboard-navigable with Tab, and a visible :focus-visible outline shows which node is currently focused — this was needed because default SVG focus styling isn't consistent across browsers.

Content Page

Technical Description

The article uses one <h1> for the page title and <h2>/<h3> for section and sub-section headings, kept in strict hierarchical order. A <nav> element with an ordered list of anchor links acts as an in-page contents menu, and each major section carries a unique id that the contents links and the "Back to Top" button target directly via href="#id". scroll-margin-top is set on each section so that clicking a contents link doesn't leave the heading hidden underneath the sticky header. Images are wrapped in <figure>/<figcaption> and styled with border, border-radius, and box-shadow to match the card language used elsewhere on the site, and a CSS Grid layout pairs two images side by side on wider screens only. I structured the content around three distances from the reader — campus, household, and community — because the spec asks for "meaningful internal navigation," which only makes sense if the sections are genuinely distinct enough to want to jump between.

Accessibility

Every image has descriptive alt text that explains what the image shows rather than restating the caption. Section headings are wired to their parent <section> via aria-labelledby, and the in-page contents list is a real <nav> landmark so assistive technology users can jump straight to it.

Challenges and Lessons Learned

One issue I ran into was that the shared .card class in the Global CSS had no padding rule, so content on every page — not just mine — was sitting flush against the card edges. Since this lived in the Global CSS rather than my own page, fixing it meant raising it with the team and editing the shared file, not working around it locally. Building the Sitemap also meant re-reading the spec closely: my first draft used a plain bullet list of links, which doesn't meet the requirement for a graphical SVG sitemap with hierarchy, hover feedback, and keyboard accessibility, so I rebuilt it from scratch as a coordinate-based SVG tree. I also found that the nav-toggle <label> in an earlier version of the shared header used role="button", which the HTML validator flags as invalid on that element — switching to a visually-hidden text label solved it without changing how the button looks or behaves for anyone.

Compliance

The pages I built contain no offensive, unlawful, or commercially exploitative material, and are limited to educational content in support of this coursework, consistent with JANET's acceptable-use conditions for connected institutions. Any external image I reference is royalty-free and credited on the Content Page under "Image Credits" rather than reproduced without permission. No personal data is collected or transmitted by my pages — the User Profile page stores information only in memory in the browser tab and never sends it to a server, so no third-party data-handling or privacy concerns arise from its use.
